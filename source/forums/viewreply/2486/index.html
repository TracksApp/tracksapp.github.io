Eventually those recurring actions moved into the home page, into the todo list and then I couldn’t use at all anymore. The error on the logs was the same.

After much experiment, trial and error, googling MySQL tutorials, I managed to fix it. Here’s how to do it for other users that come across it.

NOTE:be cautious when doing this. There’s 2 things you should know when doing this. First, you’ll be messing manually with your MySQL database and you may break something. I think there’s ways to backup it before with some DUMP commands but I don’t know about them. Second, the actions that are causing the problem will be removed completely. There’s probably some way to fix those entries other than remove but I don’t know them.

Now that you’ve been warned, on with the fix. First log in into MySQL (you’ll be asked by the MySQL root password, not the root system password. You can probably also log in as the user that you’ve set up for tracks).

mysql -u root -p 

Then, select the database that you created for tracks. In my case it was named tracks but you can check first by looking at all the available databases

mysql> SHOW databases;
+--------------------+
| Database           |
+--------------------+
| information_schema |
| drupal6            |
| mysql              |
| tracks             |
+--------------------+
mysql> USE tracks; 

By looking at the list of tables inside the database tracks, you should see something like this

mysql> SHOW tables;
+-------------------------------------+
| Tables_in_tracks                    |
+-------------------------------------+
| contexts                            |
| notes                               |
| open_id_authentication_associations |
| open_id_authentication_nonces       |
| preferences                         |
| projects                            |
| recurring_todos                     |
| schema_migrations                   |
| sessions                            |
| taggings                            |
| tags                                |
| todos                               |
| users                               |
+-------------------------------------+ 

If you look into the todos tables, you should see all the actions that would appear in your todos list with all its data (some numbers will be IDs refering to data on other tables). It should look something like this

mysql> SELECT * FROM todos;
+----+------------+------------+--------------------------------------+----------+---------------------+---------------------+---------------------+---------+---------------------+-----------+-------------------+---------------------+
| id | context_id | project_id | description                          | notes    | created_at          | due                 | completed_at        | user_id | show_from           | state     | recurring_todo_id | updated_at          |
+----+------------+------------+--------------------------------------+----------+---------------------+---------------------+---------------------+---------+---------------------+-----------+-------------------+---------------------+
| 56 |         17 |       NULL | Mogwai concert (buy tickets)         |          | 2010-10-06 13:07:45 | NULL                | NULL                |       2 | 2011-01-11 00:00:00 | deferred  |              NULL | 2010-11-04 12:15:16 |
| 72 |         18 |       NULL | throw away bad frozen cells          |          | 2010-11-09 18:13:31 | NULL                | NULL                |       2 | NULL                | active    |              NULL 
| 47 |         21 |       NULL | pay House                            |          | 2010-09-30 11:55:02 | 2010-11-19 00:00:00 | NULL                |       2 | NULL                | active    |                 2 | 2010-11-16 12:46:10 | 

You’ll noticed that some of these entries will have a non-NULL value on the collumn recurring_todo_id. If you remove these entries, it should work again (note: in my case, all of these were the problem but by looking at the post I mentioned before, it seems only actions that are on this table with this value true and not in the recurring_todos table at the same time are the problem). To remove an entry from this table with a specific value on recurring_todo_id, use the following command (replace the value 2 with the value of your problematic action. If you have more than one, run the command several times

mysql> DELETE FROM todos WHERE recurring_todo_id=2; 

Finnally, if you look at the todos table again, the entry with a recurring_todo_id value 2 whould be gone and you can exit MySQL.

mysql> SELECT * FROM todos;
+----+------------+------------+--------------------------------------+----------+---------------------+---------------------+---------------------+---------+---------------------+-----------+-------------------+---------------------+
| id | context_id | project_id | description                          | notes    | created_at          | due                 | completed_at        | user_id | show_from           | state     | recurring_todo_id | updated_at          |
+----+------------+------------+--------------------------------------+----------+---------------------+---------------------+---------------------+---------+---------------------+-----------+-------------------+---------------------+
| 56 |         17 |       NULL | Mogwai concert (buy tickets)         |          | 2010-10-06 13:07:45 | NULL                | NULL                |       2 | 2011-01-11 00:00:00 | deferred  |              NULL | 2010-11-04 12:15:16 |
| 72 |         18 |       NULL | throw away bad frozen cells          |          | 2010-11-09 18:13:31 | NULL                | NULL                |       2 | NULL                | active    |              NULL 
mysql> exit;
